## User Constraints (from CONTEXT.md)

### Locked Decisions
### App shell structure
- **D-01:** The app should use tabs from day one rather than a single-screen shell.
- **D-02:** The initial tab bar should include `Home`, `History`, and `Settings`.
- **D-03:** The Phase 1 shell should feel like the long-term product structure, not a temporary starter layout.

### Privacy framing
- **D-04:** The local-only privacy message should appear as a dedicated full-screen first-run card before the user enters the main shell.
- **D-05:** This first-run privacy screen should clearly explain that readings stay on the device and that the product is designed around privacy-first local storage.

### Dashboard empty state
- **D-06:** Before any readings exist, the dashboard should use an Apple Health-style summary shell rather than a sparse blank screen or a single guided CTA.
- **D-07:** The empty dashboard should already feel like a polished health dashboard, even without data.

### Visual direction
- **D-08:** Phase 1 should lean into a close homage to Apple Health's white, airy, soft clinical card-based look.
- **D-09:** The interface should feel calm and premium immediately, not defer the main visual identity to a later phase.

### Phase 1 behavior for not-yet-built areas
- **D-10:** The `History` tab should exist in Phase 1 and show the intended final screen structure with an empty-state message explaining that readings will appear there.
- **D-11:** The app should avoid "coming soon" messaging for the History tab; it should look like a real part of the product, just without data yet.

### Claude's Discretion
- Exact icon set for the tabs
- Specific copy wording for the privacy explanation, as long as it preserves the local-only meaning and non-diagnostic product framing
- Precise spacing, typography, and component tokens within the Apple Health-inspired direction

### Deferred Ideas (OUT OF SCOPE)
- Real reading capture and saved latest-reading content belong to Phase 2.
- Actual editable history behavior belongs to Phase 3.
- Trend visualizations and chart data belong to Phase 3.
- PDF export belongs to Phase 4.

## Summary

Phase 1 should be planned as a durable Expo Router shell, not a starter screen that gets replaced later. Expo recommends Expo Router for Expo projects, and its documented JavaScript tabs layout maps directly to the long-lived `Home` / `History` / `Settings` structure this phase requires. [CITED: https://docs.expo.dev/develop/app-navigation/] [CITED: https://docs.expo.dev/router/advanced/tabs/] The current SDK 55 starter generated by `create-expo-app` uses `expo-router/unstable-native-tabs`, but Expo documents native tabs as alpha/experimental and subject to change, so the safer foundation for this phase is the stable JavaScript `Tabs` layout with a root `Stack` and a `(tabs)` route group. [VERIFIED: create-expo-app scaffold] [CITED: https://docs.expo.dev/router/advanced/native-tabs/] [CITED: https://docs.expo.dev/router/advanced/tabs/]

The first-run privacy message should be modeled as a dedicated route that gates entry into the tab shell until the user acknowledges it once. That keeps privacy framing explicit and in-context without inventing an account or onboarding system. [ASSUMED] The acknowledgement should be persisted locally using a minimal on-device store only; do not introduce any remote config, analytics, feature flags, or update checks into the launch path for this phase. Expo Router itself is offline-first, and Expo warns that startup waits should be kept as short as possible when using splash control. [CITED: https://docs.expo.dev/router/introduction/] [CITED: https://docs.expo.dev/versions/latest/sdk/splash-screen/] [CITED: https://docs.expo.dev/eas-update/download-updates/]

History and Settings should ship as real screens in Phase 1, but empty-state driven. The planner should treat them as stable containers with final information architecture, accessibility labels, and test IDs now, while deferring any data or edit behavior to later phases. [ASSUMED] That gives later phases a fixed routing and layout contract while satisfying the requirement that the app already feels like the long-term product. [ASSUMED]

**Primary recommendation:** Build Phase 1 on Expo SDK 55 with Expo Router `Stack` + stable JavaScript `Tabs`, a dedicated first-run privacy gate persisted locally, and three production-shaped empty-state screens that require zero network to render. [VERIFIED: npm registry] [CITED: https://docs.expo.dev/router/advanced/tabs/] [CITED: https://docs.expo.dev/develop/app-navigation/] [ASSUMED]

## Architecture Patterns

### Recommended Project Structure
```text
src/
├── app/
│   ├── _layout.tsx              # Root stack, splash coordination, providers
│   ├── privacy.tsx              # First-run local-only explanation gate
│   └── (tabs)/
│       ├── _layout.tsx          # Home / History / Settings tab config
│       ├── index.tsx            # Home dashboard empty shell
│       ├── history.tsx          # Real history screen with empty state
│       └── settings.tsx         # Real settings shell with privacy copy
├── components/
│   ├── ui/                      # Card, section, list-row, stat-shell primitives
│   └── shell/                   # Empty-state illustrations / tab badges / privacy card
├── features/
│   ├── onboarding/              # First-run acknowledgement logic
│   └── readings/                # Empty selectors and future domain boundary
├── lib/
│   ├── storage/                 # Tiny local storage adapter for phase flags
│   └── theme/                   # Tokens, spacing, typography, colors
└── test/
    ├── render.tsx               # Custom RTL render helper
    └── fixtures/                # Phase fixtures and route-state helpers
```

### Pattern 1: Root Stack For Gate + Tabs
**What:** Use `src/app/_layout.tsx` as the single composition point for splash coordination, providers, and a `Stack` that contains a privacy gate route plus the `(tabs)` group. [CITED: https://docs.expo.dev/router/advanced/tabs/] [CITED: https://docs.expo.dev/versions/latest/sdk/splash-screen/]

**When to use:** Always for this phase. It gives the planner a stable place to add future modal routes without reshaping the shell. [ASSUMED]

**Example:**
```tsx
// Source: https://docs.expo.dev/router/advanced/tabs/
import { Stack } from 'expo-router';

export default function RootLayout() {
  return (
    <Stack>
      <Stack.Screen name="privacy" options={{ headerShown: false }} />
      <Stack.Screen name="(tabs)" options={{ headerShown: false }} />
    </Stack>
  );
}
```

### Pattern 2: Stable JavaScript Tabs, Not Starter Native Tabs
**What:** Use an explicit `(tabs)/_layout.tsx` with `Tabs.Screen` entries for `index`, `history`, and `settings`. [CITED: https://docs.expo.dev/router/advanced/tabs/]

**When to use:** For the permanent Phase 1 shell. JavaScript tabs are the stable documented API; native tabs are alpha and have platform limitations. [CITED: https://docs.expo.dev/router/advanced/tabs/] [CITED: https://docs.expo.dev/router/advanced/native-tabs/]

**Example:**
```tsx
// Source: https://docs.expo.dev/router/advanced/tabs/
import { Tabs } from 'expo-router';

export default function TabLayout() {
  return (
    <Tabs screenOptions={{ headerShown: false }}>
      <Tabs.Screen name="index" options={{ title: 'Home', tabBarButtonTestID: 'tab-home' }} />
      <Tabs.Screen name="history" options={{ title: 'History', tabBarButtonTestID: 'tab-history' }} />
      <Tabs.Screen name="settings" options={{ title: 'Settings', tabBarButtonTestID: 'tab-settings' }} />
    </Tabs>
  );
}
```

### Pattern 3: One-Time Privacy Gate With Local Persistence
**What:** Show a dedicated `privacy` route until a local boolean indicates the user has acknowledged the local-only explanation. [ASSUMED]

**When to use:** On first launch only. Keep the route in the app permanently so Settings can deep-link back to the privacy explanation later if needed. [ASSUMED]

**Example:**
```tsx
// Source basis: Expo Router stack layout + local-only first-run gating pattern
import { Redirect } from 'expo-router';

export default function IndexGate() {
  const hasSeenPrivacy = true; // replace with local storage read

  if (!hasSeenPrivacy) {
    return <Redirect href="/privacy" />;
  }

  return <Redirect href="/(tabs)" />;
}
```

### Pattern 4: Empty-State Driven Real Screens
**What:** Home, History, and Settings should each render a production-shaped screen scaffold even with no readings. [ASSUMED]

**When to use:** Throughout Phase 1. The app should feel complete in structure while data features are deferred. [ASSUMED]

**Implementation shape:**  
Home should render summary cards, trend placeholders, and a clear “no readings yet” explanation without a dead-end blank page. [ASSUMED]  
History should render the final list container, top summary/header treatment, and an empty-state panel that explains where readings will appear. [ASSUMED]  
Settings should render privacy/storage language and future export/privacy entry points as inactive or informational rows, not “coming soon” callouts. [ASSUMED]

### Pattern 5: Short Splash, No Network Bootstrap
**What:** Only keep the splash visible while local resources such as fonts, theme hydration, and the privacy-ack read finish. [CITED: https://docs.expo.dev/versions/latest/sdk/splash-screen/] [CITED: https://docs.expo.dev/develop/user-interface/fonts/]

**When to use:** Always. Expo says the goal should be to hide the splash as soon as possible. [CITED: https://docs.expo.dev/versions/latest/sdk/splash-screen/]

**Example:**
```tsx
// Source: https://docs.expo.dev/versions/latest/sdk/splash-screen/
import { Stack } from 'expo-router';
import * as SplashScreen from 'expo-splash-screen';
import { useEffect, useState } from 'react';

SplashScreen.preventAutoHideAsync();

export default function RootLayout() {
  const [ready, setReady] = useState(false);

  useEffect(() => {
    async function bootstrap() {
      // local font load / local storage read only
      setReady(true);
    }
    bootstrap();
  }, []);

  useEffect(() => {
    if (ready) SplashScreen.hideAsync();
  }, [ready]);

  if (!ready) return null;
  return <Stack />;
}
```

### Anti-Patterns to Avoid
- **Using `expo-router/unstable-native-tabs` as the long-term foundation:** The current starter does this, but Expo marks native tabs as alpha/experimental and subject to change. [VERIFIED: create-expo-app scaffold] [CITED: https://docs.expo.dev/router/advanced/native-tabs/]
- **Blocking launch on remote assets or services:** Phase 1 must not wait on analytics, remote fonts, remote config, or OTA update checks before rendering the shell. [CITED: https://docs.expo.dev/versions/latest/sdk/splash-screen/] [CITED: https://docs.expo.dev/eas-update/download-updates/] [ASSUMED]
- **Treating History/Settings as placeholders outside the navigator:** The tab contract should be stable now so later phases add behavior, not routes. [ASSUMED]
- **Building a custom onboarding framework:** This phase only needs a single local-only acknowledgement screen, not multi-step onboarding state. [ASSUMED]

## Common Pitfalls

### Pitfall 1: Keeping The Starter’s Native Tabs Because “Expo Chose It”
**What goes wrong:** The app inherits `expo-router/unstable-native-tabs` from the current template and treats an alpha API as the permanent shell foundation. [VERIFIED: create-expo-app scaffold] [CITED: https://docs.expo.dev/router/advanced/native-tabs/]
**Why it happens:** The SDK 55 default scaffold currently ships with `AppTabs` over unstable native tabs, so it looks like the blessed default unless you read the Router docs. [VERIFIED: create-expo-app scaffold] [CITED: https://docs.expo.dev/router/advanced/native-tabs/]
**How to avoid:** Replace the starter abstraction with an explicit `(tabs)` route group using stable JavaScript tabs before building app-specific UI. [CITED: https://docs.expo.dev/router/advanced/tabs/]
**Warning signs:** `src/components/app-tabs.tsx` survives into product code and imports `expo-router/unstable-native-tabs`. [VERIFIED: create-expo-app scaffold]

### Pitfall 2: Over-Engineering The Privacy Gate
**What goes wrong:** A single privacy explanation turns into a multi-step onboarding flow with progress state, swipes, or remote copy management. [ASSUMED]
**Why it happens:** Teams conflate “first-run screen” with onboarding infrastructure. [ASSUMED]
**How to avoid:** Keep one dedicated screen, one local acknowledgement flag, and one route decision. [ASSUMED]
**Warning signs:** More than one first-run route, or any new dependency on auth/remote config/analytics for Phase 1. [ASSUMED]

### Pitfall 3: Blocking The Splash On Non-Essential Work
**What goes wrong:** Launch time stretches because the app waits for remote fonts, update checks, or heavy initialization before hiding the splash. [CITED: https://docs.expo.dev/versions/latest/sdk/splash-screen/] [CITED: https://docs.expo.dev/eas-update/download-updates/]
**Why it happens:** `preventAutoHideAsync()` is treated as a general startup loading screen instead of a narrow readiness guard. [CITED: https://docs.expo.dev/versions/latest/sdk/splash-screen/]
**How to avoid:** Only gate on local prerequisites needed for the first frame. [CITED: https://docs.expo.dev/versions/latest/sdk/splash-screen/] [ASSUMED]
**Warning signs:** Bootstrap code reaches network, waits on OTA checks, or shows a blank screen after the splash disappears. [CITED: https://docs.expo.dev/eas-update/download-updates/] [ASSUMED]

### Pitfall 4: Shipping Fake Tabs
**What goes wrong:** History and Settings are tabs in name only, implemented as centered placeholder text or “coming soon” messaging. [ASSUMED]
**Why it happens:** The phase is scoped to shell work, so teams underinvest in empty-state design. [ASSUMED]
**How to avoid:** Plan final screen skeletons now: top spacing, headers, cards, supporting copy, and navigational affordances. [ASSUMED]
**Warning signs:** Empty tabs lack headers, lack real spacing tokens, or read like backlog notes rather than product copy. [ASSUMED]

### Pitfall 5: Starting In Expo Go And Deferring Dev Builds Too Long
**What goes wrong:** The team cannot validate splash behavior, native assets, or future SQLite/native-module additions with production parity. [CITED: https://docs.expo.dev/develop/development-builds/introduction/] [CITED: https://docs.expo.dev/versions/latest/sdk/splash-screen/]
**Why it happens:** Expo Go is convenient early, but Expo documents it as a playground and not a production-grade environment. [CITED: https://docs.expo.dev/develop/development-builds/introduction/]
**How to avoid:** Bootstrap with Expo, but schedule the first Android development build in Wave 0 or Wave 1 once the shell compiles. [CITED: https://docs.expo.dev/develop/development-builds/introduction/] [ASSUMED]
**Warning signs:** Review decisions are made from Expo Go screenshots only. [ASSUMED]

### Splash Coordination For Local Bootstrap
```tsx
// Source: https://docs.expo.dev/versions/latest/sdk/splash-screen/
import { Stack } from 'expo-router';
import * as SplashScreen from 'expo-splash-screen';
import { useEffect, useState } from 'react';

SplashScreen.preventAutoHideAsync();

export default function RootLayout() {
  const [isReady, setIsReady] = useState(false);

  useEffect(() => {
    async function doAsyncStuff() {
      try {
        // local async work only
      } finally {
        setIsReady(true);
      }
    }

    doAsyncStuff();
  }, []);

  useEffect(() => {
    if (isReady) SplashScreen.hideAsync();
  }, [isReady]);

  if (!isReady) return null;
  return <Stack />;
}
```
